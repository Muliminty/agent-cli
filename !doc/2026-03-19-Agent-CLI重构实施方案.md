# Agent CLI 重构实施方案

> 更新时间：2026-03-19
> 目标：把当前的“功能很多但闭环不稳”的半成品，收敛成一个可持续运行、可监控、可验证、可迭代的本地 Agent Orchestrator。

## 1. 这份方案解决什么问题

当前项目已经具备以下基础：

- CLI 命令体系
- Web 控制台
- WebSocket 实时通信
- 多 AI provider 接入
- 进度跟踪
- 测试框架
- 初步的 Agent 抽象

但它仍然不是一个真正稳定的“主管 Agent 驱动执行 Agent 自动完成开发任务”的系统，主要问题在于：

- 模块很多，但没有围绕一个清晰的任务会话状态机收敛
- `CLI -> API -> Web` 已有桥接，但“任务执行 -> 验证 -> 再决策”的闭环不够明确
- 代码里“能力”很多，“编排”不足
- README / PROGRESS / TODO 对完成度的描述偏乐观，和当前真实测试状态不完全一致
- 测试并未完全绿灯，工程状态还没有进入“可以放心继续扩展”的阶段

这份方案的核心不是继续加功能，而是先把系统重构成一个明确的 Supervisor/Worker/Verifier 架构，并围绕“任务会话”统一 CLI、Web、测试和 AI 执行流程。

## 2. 当前项目的真实判断

### 2.1 当前最接近的定位

当前项目更像：

**面向长任务开发工作流的本地 Agent 控制台原型**

而不是：

**一个已经稳定可用的 Claude Code / Codex 风格自动开发编排器**

### 2.2 当前最主要的差距

项目已经有：

- Agent 基类：`src/core/agent/base.ts`
- 编码 Agent：`src/core/agent/coder.ts`
- 测试管理器：`src/core/test/test-manager.ts`
- Web API：`src/server/api/cli-commands.ts`
- Web 控制台：`src/server/index.ts` + `public/`

但还缺：

- 明确的 `TaskSession` 模型
- Supervisor 持续观察执行状态的机制
- 对“完成 / 卡住 / 跑偏 / 失败”的统一判定规则
- Verifier 独立验收层
- 针对 Claude Code 这类外部执行器的稳定进程桥接
- 可恢复的长任务状态持久化

### 2.3 当前已确认的问题

目前需要明确承认以下工程现实：

- 测试不是全绿
- 文档状态和真实代码状态有偏差
- Web API 更像“把 CLI 包成 HTTP”，而不是“任务编排服务”

已观察到的具体问题：

1. `src/server/services/ai-service.ts` 在 Jest 环境下存在模块解析问题  
   报错：`Cannot find module '../../utils/logger.js'`
2. `src/config/__tests__/schema.test.ts` 断言与当前默认配置不一致  
   表现为 provider 默认字段新增后，测试期望未同步

这两个问题说明项目当前最优先的事不是继续扩展，而是先恢复“真实可维护状态”。

## 3. 最终要做成什么样

## 3.1 最终产品定义

把 Agent CLI 做成一个本地运行的开发编排器，支持以下工作流：

1. 用户通过 CLI 或 Web UI 创建一个开发任务
2. 系统创建一个任务会话 `TaskSession`
3. Supervisor 为任务拆解阶段目标
4. Worker 执行编码、命令运行、测试、修复等动作
5. Verifier 根据测试结果、构建结果、规则检查和验收清单判断结果
6. 如果未完成，Supervisor 自动生成下一轮更聚焦的执行指令
7. 整个过程在 Web UI 中可见，可暂停、继续、重试、接管
8. 最终在满足验收条件后，将任务标记为完成，并生成报告

## 3.2 目标效果

完成后的系统应该具备下面这些清晰特征：

### CLI 体验

- `agent-cli run <task>` 可以直接启动一个完整任务会话
- CLI 能显示当前阶段、执行轮次、最近一次验证结果、失败原因、下一步计划
- CLI 支持 `pause / resume / retry / cancel / inspect`

### Web UI 体验

- Web UI 不是终端套壳，而是“任务控制台”
- 能看到每个任务的状态、轮次、当前动作、最近一次命令输出、测试结果、验收结论
- 能单独查看 Supervisor 判断、Worker 日志、Verifier 结论
- 能手动接管某一轮，或重新下发一条更明确的指令

### 执行能力

- 支持把 Claude Code / Codex CLI / 未来其他 coder agent 作为 Worker Adapter
- 支持监控外部进程
- 支持流式输出
- 支持判断超时、卡死、异常退出、重复失败

### 验收能力

- 不靠“模型说完成了”作为完成依据
- 必须基于真实证据判定：测试、构建、lint、文件变更、规则检查
- Verifier 可输出结构化失败原因

### 长任务稳定性

- 任务状态持久化
- 页面刷新后能恢复
- 进程退出后仍能查看历史
- 支持中断恢复

## 3.3 最终产品体验定义

这是本项目后续实现时必须对齐的最终体验目标。

### 用户主流程

用户体验应当是：

1. 用户先在 Web 界面中和主管 Agent 讨论需求
2. 系统把讨论内容沉淀成结构化文档
3. 用户确认或编辑文档
4. 用户点击开始执行
5. Claude Code 这类执行 Agent 在后台持续工作
6. 用户主要通过 Web 查看执行情况，而不是一直参与对话
7. 只有在关键分歧、风险升级、需要授权或验收失败时，系统才回来打扰用户
8. 最终交付成品、执行报告、测试结果与遗留风险说明

换句话说，产品目标不是“一个更强的聊天界面”，而是：

**一个可以先讨论、再定稿、再长时间自动推进任务的本地开发任务驾驶舱。**

### 最终用户感受到的效果

对用户来说，理想体验应该是：

- 先把事情和主管 Agent 聊清楚
- 把结果沉淀成正式文档
- 文档可以继续修改，不会被聊天历史绑死
- 点击开始后，执行 Agent 自己持续工作
- 用户可以随时打开 Web 查看任务进展
- 不需要一直盯着，只在必要时介入
- 最后看到成品、报告和风险说明

这部分体验必须被视为“产品核心”，而不是可选增强项。

## 3.4 Web 端必须具备的核心能力

后续 Web 端不应定位为“终端套壳”或“CLI 的远程按钮面板”，而应定位为：

**任务讨论、文档沉淀、执行监控、人工接管四位一体的任务驾驶舱。**

Web 端至少必须具备以下能力：

### A. 讨论区

用于用户与主管 Agent 讨论需求、范围、边界、风险和验收标准。

要求：

- 支持连续对话
- 支持主管 Agent 自动补齐缺失信息
- 能从对话中提炼结构化内容
- 讨论结果能同步沉淀到文档区

### B. 文档区

用于承载系统的“单一事实来源”。

至少包含以下文档对象：

- 需求文档
- 执行计划
- 验收标准
- 风险与待确认项
- 最终交付报告

要求：

- 文档必须可编辑
- 支持 AI 生成初稿
- 支持用户手动修改
- 支持版本化
- 支持将某一版本设为“当前执行依据”

必须明确一点：

**执行层应尽量基于当前文档版本工作，而不是直接依赖整段聊天历史。**

### C. 执行监控区

用于观察 Claude Code 或其他 Worker 当前正在做什么。

必须能看到：

- 当前任务状态
- 当前阶段
- 当前轮次
- 当前 Worker
- 当前动作
- 最近一次测试结果
- 最近一次失败原因
- 下一步计划
- 执行日志和流式输出

注意：

默认展示应以“状态理解”为主，而不是直接把一坨终端日志扔给用户。

### D. 控制区

用于用户在必要时介入任务流程。

至少支持：

- 暂停
- 恢复
- 取消
- 重试当前轮
- 重新规划
- 编辑文档后继续执行
- 人工接管

## 3.5 推荐的 Web 信息架构

建议最终采用三栏式或等价布局：

### 左栏：任务与会话

- 项目列表
- 任务列表
- 会话状态
- 历史记录

### 中栏：讨论与执行

建议至少包含两个主视图：

- `讨论`
- `执行流`

讨论视图用于和主管 Agent 对齐需求。  
执行流视图用于查看 Claude Code 的轮次推进、事件时间线和实时输出。

### 右栏：文档与状态

用于展示和编辑：

- 需求文档
- 执行计划
- 验收标准
- 风险清单
- 当前摘要

## 3.6 文档驱动执行原则

后续实现必须遵守文档驱动原则。

### 原则一：聊天用于探索，文档用于定稿

聊天负责澄清想法、追问细节、发现风险。  
文档负责固化当前共识，成为后续执行依据。

### 原则二：文档必须可编辑

如果文档不可编辑，系统会退化回“只能靠聊天修正需求”的状态，最终导致：

- 需求漂移
- 历史混乱
- 执行层理解不稳定

### 原则三：执行绑定文档版本

建议每次正式执行都绑定一个文档版本，例如：

- `spec v1`
- `plan v2`

如果用户修改文档，应允许系统提示：

- 是否基于新版本重新规划
- 是否继续当前执行
- 是否终止并重开会话

## 3.7 执行监控的展示原则

后续实现时，不建议用单纯的“百分比”来代表进度。

更有价值的展示应包括：

- 当前阶段
- 当前轮次
- 最近完成的里程碑
- 当前阻塞点
- 最近失败原因
- 最近验证结论
- 下一步动作

也就是说，Web 上展示的是：

**系统当前对任务状态的理解**

而不仅仅是：

**终端输出流**

## 3.8 何时应该打扰用户

目标体验不是“永远不打扰用户”，而是“只在值得打扰的时候打扰”。

系统应优先自动推进，但在以下场景必须升级为用户确认：

- 需求边界发生变化
- 出现高风险改动
- 涉及破坏性操作
- 多个方案成本和后果差异明显
- 连续多轮执行失败
- 验收标准无法满足

这部分是产品体验成功与否的关键。

真正优秀的体验不是“完全自动”，而是：

**前台少打扰，后台长执行，关键节点精准唤回用户。**

## 4. 推荐的目标架构

## 4.1 架构原则

后续重构必须遵守以下原则：

1. 先统一任务状态机，再扩展功能
2. 先把 CLI / Web / AI 调用都变成围绕 `TaskSession` 工作，再谈更多命令
3. Worker 负责执行，不负责证明自己完成
4. Verifier 独立存在，不和 Worker 混在一起
5. Web 层只展示和控制，不负责核心编排逻辑

## 4.2 目标分层

### A. Domain 层

负责核心状态与规则：

- `TaskSession`
- `TaskStep`
- `TaskPhase`
- `TaskStatus`
- `ExecutionAttempt`
- `VerificationResult`
- `CompletionCriteria`

### B. Orchestrator 层

负责调度闭环：

- `Supervisor`
- `WorkerManager`
- `Verifier`
- `SessionScheduler`
- `RecoveryManager`

### C. Adapter 层

负责对接外部系统：

- `ClaudeCodeAdapter`
- `CodexAdapter`
- `ShellCommandAdapter`
- `TestRunnerAdapter`
- `GitAdapter`

### D. Interface 层

负责与用户交互：

- CLI
- REST API
- WebSocket
- Web UI

## 4.3 任务状态机建议

建议统一任务状态如下：

- `queued`
- `planning`
- `running`
- `verifying`
- `blocked`
- `waiting_input`
- `retrying`
- `completed`
- `failed`
- `cancelled`

同时为每次执行记录 attempt：

- `attemptNumber`
- `workerType`
- `promptSummary`
- `startedAt`
- `endedAt`
- `exitCode`
- `commandSummary`
- `artifacts`
- `verificationSummary`

## 6. 关键重构策略

## 6.1 第一优先：统一“任务会话”概念

现在项目的很多能力是散的，需要先收拢成一个任务会话模型。

建议增加核心对象：

- `TaskSession`
- `TaskGoal`
- `ExecutionPlan`
- `ExecutionAttempt`
- `VerificationReport`

每个 CLI 命令、Web 控制台动作、后台任务都应尽量落到这个模型上，而不是各自维护状态。

## 6.2 第二优先：引入 Supervisor / Worker / Verifier 三层

### Supervisor

职责：

- 接收用户目标
- 决定当前阶段目标
- 为 Worker 生成本轮执行任务
- 根据 Verifier 结果决定下一轮动作
- 判断暂停、升级人工接管、最终完成

### Worker

职责：

- 执行一次具体任务
- 调 Claude Code / Codex / shell / test runner
- 返回原始输出和结构化元信息

### Verifier

职责：

- 审查当前结果
- 基于外部证据决定是否通过
- 形成结构化结论，而不是纯文本描述

这是整个项目从“工具集合”升级为“编排系统”的分水岭。

## 6.3 第三优先：把 CLI 命令 API 化升级为 Session API

当前的 `src/server/api/cli-commands.ts` 更像命令转发层。

建议改造方向：

- 保留命令调用能力
- 但 Web 层主要围绕 Session API 工作

例如新增：

- `POST /api/sessions`
- `GET /api/sessions/:id`
- `POST /api/sessions/:id/pause`
- `POST /api/sessions/:id/resume`
- `POST /api/sessions/:id/retry`
- `POST /api/sessions/:id/cancel`
- `GET /api/sessions/:id/events`

WebSocket 也围绕 session 广播事件，而不是只广播零散 operation。

## 6.4 第四优先：真正引入外部执行器适配层

如果目标是让系统“监控 Claude Code 并驱动其循环开发”，那就必须把外部执行器抽象为 Adapter。

每个 Worker Adapter 至少需要统一接口：

- `start()`
- `sendInput()`
- `streamOutput()`
- `getStatus()`
- `interrupt()`
- `terminate()`
- `collectArtifacts()`

这样后面就不只是 Claude Code，还能接 Codex CLI、Aider、shell script 等执行器。

## 6.5 第五优先：把“完成”定义成规则，不是文案

必须引入完成准则对象：

- 测试是否通过
- 构建是否通过
- lint 是否通过
- 是否存在指定输出文件
- 是否满足任务说明中的硬性约束
- 是否触发人工确认点

只有当 Verifier 输出 `pass`，Supervisor 才能把 session 标记为 completed。

## 7. 推荐实施阶段

## 阶段 0：稳定现状，恢复可信工程基线

目标：把项目从“看起来功能很多”拉回“真实可维护”。

任务：

1. 修复当前失败测试
2. 检查 Jest + ESM + `.js` import 约定
3. 更新 README / PROGRESS / TODO 中不准确的完成度表述
4. 明确哪些功能只是原型，哪些功能已可用
5. 保证 `npm test` 回到稳定通过

阶段完成标准：

- 测试全绿
- 文档与代码状态一致
- 能清楚回答“这个项目现在到底能做什么”

## 阶段 1：引入 TaskSession 核心模型

目标：统一所有后台任务的状态表达。

任务：

1. 新增 `TaskSession` 类型与存储
2. 设计状态机与事件格式
3. 把现有任务管理和 WebSocket 事件接入 Session
4. 增加最小的 Session API

阶段完成标准：

- 新任务可以创建 Session
- Session 状态可查询
- Web UI 能显示真实任务生命周期

## 阶段 2：落地 Supervisor / Worker / Verifier 闭环

目标：实现真正的自动迭代执行机制。

任务：

1. 实现 Supervisor 基础调度器
2. 把现有 `CoderAgent` 的部分职责下沉为 Worker
3. 新增 Verifier 层
4. 让系统支持：
   - 执行一轮
   - 验证一轮
   - 失败后再规划下一轮

阶段完成标准：

- 一条任务可自动经历多轮执行
- 每一轮都能产出结构化结果
- 失败不是简单报错，而是进入 retry / blocked / waiting_input

## 阶段 3：接入 Claude Code / 外部 coder agent

目标：让系统真的能把外部编码 Agent 当 Worker 使用。

任务：

1. 实现 `ClaudeCodeAdapter`
2. 统一 stdout / stderr / 退出码 / 超时 / 中断处理
3. 记录每轮执行的输入摘要、输出摘要和 artifacts
4. 增加超时、卡死、重复失败检测

阶段完成标准：

- 能启动一个外部 coder session
- 能监控其执行状态
- 能在失败后自动进入下一轮

## 阶段 4：Web UI 从“控制台”升级为“任务驾驶舱”

目标：让 UI 适合长任务观察与接管。

任务：

1. Session 列表与详情页
2. 轮次时间线
3. Worker 输出流式面板
4. Verifier 结果面板
5. 手动暂停、恢复、重试、接管

阶段完成标准：

- Web UI 能完整观察一个任务闭环
- 用户能不进终端也知道系统现在在做什么

## 阶段 5：增强验收与恢复能力

目标：让系统可以更长期稳定运行。

任务：

1. 中断恢复
2. Session 持久化
3. 验收模板化
4. 自动生成交付报告
5. 加入更多测试覆盖

阶段完成标准：

- 任务失败后可恢复
- 页面刷新后状态不丢
- 报告足够支持复盘与交接

## 8. 对现有文件的改造建议

以下不是第一轮必须全部改，但建议作为新线程中的重点关注对象。

### 8.1 `src/core/agent/base.ts`

现状：

- 是一个不错的 Agent 基类
- 但更偏“单体执行器抽象”

建议：

- 保留基础生命周期能力
- 不再让它承担全部编排职责
- 明确它在新架构里更接近 Worker 基础类，而不是整个 orchestrator 的中心

### 8.2 `src/core/agent/coder.ts`

现状：

- 把功能选择、代码修改、测试、进度更新、提交都耦合在一个类里

建议：

- 拆成更清晰的责任边界
- 保留其内部通用逻辑
- 但把“功能选择”和“完成判定”上移到 Supervisor / Verifier

### 8.3 `src/server/api/cli-commands.ts`

现状：

- 是 Web 对 CLI 的桥接层

建议：

- 保留作为兼容接口
- 但新增更高优先级的 Session API
- 后续 UI 不再围绕“执行命令”组织，而是围绕“任务会话”组织

### 8.4 `src/core/test/test-manager.ts`

现状：

- 更像一个测试执行器

建议：

- 保留作为底层测试能力
- 在其外层增加 `TestVerifier`
- 让测试结果不只是“展示”，而是参与完成判定

### 8.5 `src/server/index.ts` 与 `public/`

现状：

- 已经具备 Web 服务和前端仪表板基础

建议：

- 不推翻重做
- 在现有 WebSocket / API 基础上重构数据模型
- UI 优先补足 Session 视图、事件时间线和状态可视化

## 9. 下一线程建议的实际执行顺序

如果下个线程直接进入开发，建议严格按下面顺序推进：

1. 先修测试和文档一致性问题
2. 再定义 `TaskSession`、状态机、事件模型
3. 再把 WebSocket 与任务管理统一到 Session
4. 再实现最小 Supervisor / Verifier
5. 再接入外部执行器 Adapter
6. 最后升级 Web UI 和补测试

不要一开始就急着：

- 加更多 CLI 命令
- 重写前端样式
- 扩更多 provider
- 做太早的性能优化

这些都不是当前瓶颈。

## 10. 下一线程的明确交付目标

新线程里，第一阶段完成后，项目应该达到以下“肉眼可感知”的效果：

### 可执行效果

- 可以创建一个任务会话
- 可以从 CLI 或 Web 发起一次真实任务
- 系统能显示任务当前状态
- 一轮执行失败后，不是直接结束，而是给出结构化失败原因
- 系统能进入下一轮或等待人工接管

### 可观测效果

- Web UI 中能看到：
  - 任务状态
  - 当前轮次
  - 当前 Worker
  - 最新测试结果
  - 最近一次失败原因
  - 下一步计划

### 可验证效果

- 测试重新回到稳定通过
- 文档和真实能力一致
- 至少有一个端到端演示路径跑通

## 11. 最终完成态描述

项目最终完成后，它应该不是一个“能执行很多命令的 CLI”，而是一个：

**本地开发任务编排平台**

它应该做到：

- 能把一个大任务拆成阶段推进
- 能驱动 Claude Code / Codex CLI 这类执行器工作
- 能观察执行过程
- 能判断是否真的完成
- 能在失败后自动重试或升级人工接管
- 能通过 CLI 和 Web UI 两种方式操作
- 能保存会话、查看历史、复盘问题

换句话说，最终效果不是“命令更多”，而是：

**系统终于开始像一个真正的主管 Agent。**
